# Библиотека QA-промптов

Эти промпты предназначены для учебных QA-задач. Их нельзя использовать как замену проверки требований и результата. Каждый ответ ИИ нужно ревьюить как черновик.

Общее правило для всех промптов:

```text
Если данных не хватает, напиши «Недостаточно данных».
Если делаешь предположение, пометь его как «Допущение».
Не выдумывай требований, которых нет во входных данных.
```

## 1. Анализ требований

**Задача:** разобрать требование простыми словами и выделить проверяемые правила.

**Когда использовать:** когда требование кажется длинным, сложным или плохо структурированным.

**Промпт:**

```text
Ты помогаешь начинающему QA анализировать требование.

Входные данные:
[вставь требование]

Задача:
1. Объясни требование простыми словами.
2. Выдели бизнес-правила.
3. Выдели ограничения и условия.
4. Отдельно перечисли, что можно проверить тестами.
5. Отдельно перечисли, каких данных не хватает.

Ограничения:
- не выдумывай требований;
- если данных нет, пиши «Недостаточно данных»;
- если делаешь предположение, помечай «Допущение».

Формат:
- Краткое объяснение
- Бизнес-правила
- Ограничения
- Что тестировать
- Недостаточно данных
- Допущения
```

**Что проверить в ответе:**

- ИИ не добавил функции, которых нет в требовании.
- Правила можно превратить в проверки.
- Неясности вынесены отдельно.

**Типичные ошибки ИИ:**

- Додумывает поведение системы.
- Смешивает факты и предположения.
- Пропускает ограничения.

## 2. Поиск неоднозначностей

**Задача:** найти неясные места и подготовить вопросы к аналитику или заказчику.

**Когда использовать:** перед составлением тест-кейсов по неполному требованию.

**Промпт:**

```text
Ты помогаешь QA найти неоднозначности в требовании.

Требование:
[вставь требование]

Задача:
Найди места, которые могут быть поняты по-разному или мешают тестированию.

Для каждого пункта укажи:
- фрагмент требования;
- в чем проблема;
- какой риск для тестирования;
- вопрос к аналитику.

Ограничения:
- не исправляй требование сам;
- не выбирай правильную трактовку без подтверждения;
- помечай «Недостаточно данных», если информации нет.
```

**Что проверить в ответе:**

- Вопросы относятся к исходному требованию.
- ИИ не превращает вопросы в утверждения.
- Риски понятны начинающему QA.

**Типичные ошибки ИИ:**

- Предлагает решение вместо вопроса.
- Драматизирует низкие риски.
- Формулирует слишком общие вопросы.

## 3. Генерация тест-кейсов

**Задача:** получить черновик тест-кейсов по требованию.

**Когда использовать:** после первичного самостоятельного анализа требования.

**Промпт:**

```text
Ты помогаешь начинающему QA составить тест-кейсы.

Требование:
[вставь требование]

Задача:
Составь тест-кейсы с позитивными, негативными и граничными проверками.

Ограничения:
- не добавляй функций, которых нет в требовании;
- если данных не хватает, укажи «Недостаточно данных»;
- помечай «Допущение» для предположений;
- ожидаемый результат должен быть проверяемым.

Формат:
Таблица: ID, Название, Предусловия, Шаги, Тестовые данные, Ожидаемый результат, Тип проверки.

В конце добавь список того, что QA должен проверить в твоем ответе.
```

**Что проверить в ответе:**

- Есть позитивные и негативные проверки.
- Есть граничные значения.
- Нет неподтвержденных функций.
- Ожидаемые результаты конкретны.

**Типичные ошибки ИИ:**

- Добавляет UI-детали без основания.
- Пропускает границы.
- Делает слишком общие expected result.

## 4. Генерация чек-листа

**Задача:** получить компактный чек-лист проверок.

**Когда использовать:** когда нужен быстрый набор идей для exploratory testing или smoke-проверки.

**Промпт:**

```text
Ты помогаешь QA составить чек-лист.

Контекст:
[что тестируем]

Входные данные:
[требование или описание фичи]

Задача:
Составь чек-лист проверок. Раздели проверки по группам.

Ограничения:
- не выдумывай требований;
- добавь позитивные, негативные и граничные проверки;
- если данных не хватает, напиши «Недостаточно данных»;
- не делай чек-лист слишком длинным.

Формат:
- Группа проверок
  - Проверка
```

**Что проверить в ответе:**

- Чек-лист покрывает основные риски.
- Группы логичны.
- Нет лишних функций.

**Типичные ошибки ИИ:**

- Делает чек-лист слишком общим.
- Дублирует проверки разными словами.
- Добавляет нерелевантные проверки безопасности или производительности.

## 5. Улучшение баг-репорта

**Задача:** структурировать плохой баг-репорт без выдумывания фактов.

**Когда использовать:** когда описание бага неполное или плохо оформлено.

**Промпт:**

```text
Ты помогаешь QA улучшить баг-репорт.

Исходное описание:
[вставь описание]

Задача:
Преобразуй описание в структурированный баг-репорт.

Ограничения:
- не выдумывай факты;
- неизвестные поля заполняй как «Недостаточно данных»;
- отделяй факты от предположений;
- составь вопросы к автору бага.

Формат:
- Summary
- Environment
- Preconditions
- Steps to reproduce
- Actual result
- Expected result
- Severity
- Priority
- Attachments/logs
- Missing information
- Questions to reporter
```

**Что проверить в ответе:**

- Не придуманы браузер, устройство, severity, priority.
- Есть список недостающих данных.
- Actual result отделен от expected result.

**Типичные ошибки ИИ:**

- Уверенно назначает severity.
- Придумывает точный текст ошибки.
- Убирает важную неопределенность.

## 6. Подбор тестовых данных

**Задача:** предложить безопасные учебные тестовые данные.

**Когда использовать:** для проверки форм, валидации и граничных условий.

**Промпт:**

```text
Ты помогаешь QA подобрать тестовые данные.

Поле или сценарий:
[описание поля или сценария]

Правила:
[валидация, ограничения, формат]

Задача:
Предложи набор тестовых данных для позитивных, негативных и граничных проверок.

Ограничения:
- не используй реальные персональные данные;
- используй только учебные примеры;
- если правило не указано, напиши «Недостаточно данных»;
- для каждого значения объясни, что оно проверяет.

Формат:
Таблица: Значение, Тип проверки, Что проверяем, Ожидаемый результат.
```

**Что проверить в ответе:**

- Данные не похожи на реальные персональные данные.
- Есть валидные и невалидные значения.
- Есть границы.

**Типичные ошибки ИИ:**

- Использует реальные домены или персональные шаблоны.
- Не объясняет назначение данных.
- Пропускает пустые значения.

## 7. Классы эквивалентности и границы

**Задача:** применить базовый тест-дизайн к полю или правилу.

**Когда использовать:** когда у требования есть диапазоны, длины, ограничения или форматы.

**Промпт:**

```text
Ты помогаешь начинающему QA применить тест-дизайн.

Правило:
[вставь правило]

Задача:
Определи классы эквивалентности и граничные значения.

Ограничения:
- не добавляй правил, которых нет во входных данных;
- если границы не указаны, напиши «Недостаточно данных»;
- помечай предположения как «Допущение».

Формат:
- Валидные классы
- Невалидные классы
- Граничные значения
- Минимальный набор проверок
- Что QA должен перепроверить
```

**Что проверить в ответе:**

- Границы соответствуют требованию.
- Есть значения ниже, на границе и выше границы.
- Нет лишних правил.

**Типичные ошибки ИИ:**

- Неверно выбирает соседние значения.
- Подменяет классы конкретными тест-кейсами без объяснения.
- Додумывает формат.

## 8. Таблица решений

**Задача:** разложить комбинации условий и результатов.

**Когда использовать:** когда поведение зависит от нескольких условий.

**Промпт:**

```text
Ты помогаешь QA составить таблицу решений.

Описание логики:
[вставь описание]

Задача:
Выдели условия и возможные результаты. Составь таблицу решений.

Ограничения:
- не выдумывай условий;
- если результат для комбинации не указан, напиши «Недостаточно данных»;
- помечай «Допущение» для предположений;
- предложи минимальный набор тестов по таблице.

Формат:
- Условия
- Действия/результаты
- Таблица решений
- Минимальный набор тестов
- Вопросы к требованиям
```

**Что проверить в ответе:**

- Все условия взяты из требования.
- Комбинации не противоречат друг другу.
- Неизвестные результаты не придуманы.

**Типичные ошибки ИИ:**

- Строит слишком большую таблицу без пользы.
- Придумывает отсутствующие результаты.
- Пропускает конфликтующие условия.

## 9. Pairwise-идеи

**Задача:** получить идеи для сокращения комбинаций параметров.

**Когда использовать:** когда параметров много и полный перебор слишком большой.

**Промпт:**

```text
Ты помогаешь QA подготовить идеи для pairwise-тестирования.

Параметры и значения:
[перечисли параметры и значения]

Задача:
Предложи компактный набор комбинаций для проверки парных сочетаний.

Ограничения:
- не добавляй новые параметры;
- если значения параметров неполные, напиши «Недостаточно данных»;
- укажи, что результат является черновиком и требует проверки QA;
- отдельно перечисли ограничения подхода.

Формат:
Таблица комбинаций и список рисков.
```

**Что проверить в ответе:**

- Все параметры взяты из входных данных.
- Комбинации не содержат невозможных значений.
- ИИ не обещает полное покрытие всех рисков.

**Типичные ошибки ИИ:**

- Называет набор pairwise без проверки покрытия.
- Создает невозможные комбинации.
- Игнорирует бизнес-риски.

## 10. SQL-пояснение

**Задача:** объяснить SQL-запрос или помочь сформулировать безопасную идею запроса.

**Когда использовать:** когда начинающему QA нужно понять, что делает простой запрос.

**Промпт:**

```text
Ты помогаешь начинающему QA понять SQL.

SQL-запрос или задача:
[вставь запрос или задачу]

Контекст таблиц:
[если есть, вставь названия таблиц и полей]

Задача:
Объясни простыми словами, что делает запрос или как подойти к задаче.

Ограничения:
- не выдумывай схему базы данных;
- если не хватает таблиц или полей, напиши «Недостаточно данных»;
- не предлагай изменять данные, если задача только на чтение;
- помечай предположения как «Допущение».

Формат:
- Простое объяснение
- Какие данные нужны
- Возможные риски
- Что QA должен уточнить
```

**Что проверить в ответе:**

- ИИ не придумал таблицы и поля.
- Нет опасных операций изменения данных без необходимости.
- Объяснение понятно новичку.

**Типичные ошибки ИИ:**

- Выдумывает схему.
- Предлагает `UPDATE` или `DELETE` без запроса.
- Уверенно объясняет неизвестные поля.

## 11. API-тестирование

**Задача:** получить идеи проверок для API по описанию endpoint.

**Когда использовать:** когда есть описание метода, параметров и ожидаемых ответов.

**Промпт:**

```text
Ты помогаешь начинающему QA придумать API-проверки.

Описание API:
[метод, URL, параметры, тело запроса, ответы]

Задача:
Предложи проверки для API.

Ограничения:
- не выдумывай поля и коды ответов;
- если данных не хватает, напиши «Недостаточно данных»;
- включи позитивные, негативные и граничные проверки;
- отдельно укажи вопросы к документации.

Формат:
Таблица: ID, Проверка, Запрос/данные, Ожидаемый ответ, Что уточнить.
```

**Что проверить в ответе:**

- Коды ответов соответствуют документации.
- Поля запроса не выдуманы.
- Есть негативные сценарии.

**Типичные ошибки ИИ:**

- Добавляет стандартные HTTP-коды без основания.
- Придумывает авторизацию.
- Не отделяет контракт API от предположений.

## 12. Ретроспектива после тестирования

**Задача:** структурировать выводы после тестирования.

**Когда использовать:** после выполнения чек-листа или тест-кейсов.

**Промпт:**

```text
Ты помогаешь QA подвести итоги тестирования.

Входные данные:
- что тестировали: [описание]
- что проверили: [список]
- что нашли: [баги или наблюдения]
- что не успели: [если есть]

Задача:
Сформируй краткую ретроспективу тестирования.

Ограничения:
- не скрывай непроверенные области;
- не делай вывод «качество хорошее», если данных недостаточно;
- помечай «Недостаточно данных» там, где нет фактов;
- предложи улучшения процесса.

Формат:
- Что проверено
- Что найдено
- Что осталось непроверенным
- Риски
- Что улучшить в следующий раз
```

**Что проверить в ответе:**

- Непроверенные области не замаскированы.
- Выводы основаны на фактах.
- Риски сформулированы аккуратно.

**Типичные ошибки ИИ:**

- Делает слишком оптимистичный вывод.
- Превращает наблюдения в доказанные баги.
- Игнорирует ограничения тестирования.

## 13. Самопроверка результата ИИ

**Задача:** попросить ИИ проверить собственный ответ по QA-критериям.

**Когда использовать:** после получения любого QA-артефакта от ИИ.

**Промпт:**

```text
Проведи самопроверку своего предыдущего ответа как QA-ревьюер.

Проверь:
1. Ответ опирается только на входные данные?
2. Есть ли выдуманные требования?
3. Есть ли непомеченные допущения?
4. Есть ли «Недостаточно данных» там, где информации не хватает?
5. Покрыты ли позитивные, негативные и граничные сценарии?
6. Ожидаемые результаты проверяемы?
7. Есть ли риск для безопасности или приватности?
8. Что QA должен перепроверить вручную?

Формат:
Таблица: Критерий, Статус, Комментарий, Что исправить.
```

**Что проверить в ответе:**

- ИИ реально нашел слабые места, а не просто похвалил себя.
- Исправления конкретны.
- Остались пункты для ручной проверки QA.

**Типичные ошибки ИИ:**

- Ставит себе «все хорошо» без анализа.
- Не замечает собственные допущения.
- Предлагает исправления, которые снова выдумывают факты.

## Как использовать библиотеку на занятии

- Не давать студентам все промпты сразу в начале занятия.
- На практике 1 использовать промпт про тест-кейсы и самопроверку.
- На практике 2 использовать промпт про баг-репорт.
- На практике 3 использовать идеи из промптов про анализ требований и самопроверку.
- После занятия предложить студентам выбрать 2-3 промпта для домашней тренировки.
