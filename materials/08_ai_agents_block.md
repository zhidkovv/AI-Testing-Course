# Блок про ИИ-агентов

## Зачем этот блок в занятии

Начинающему QA важно понимать разницу между обычным ИИ-чатом и агентным workflow. Без этого слово «агент» легко воспринимается как рекламный термин или как «чат, который умнее». Для тестировщика полезнее другая рамка: агент - это повторяемый процесс с целью, инструментами, действиями, проверками и контролем человека.

Главное правило остается тем же:

```text
ИИ-агент может ускорить работу, но QA отвечает за проверку результата.
```

## Простое определение

ИИ-агент - это ИИ-система или рабочий режим, которому дают цель, контекст и доступные инструменты. Агент планирует шаги, выполняет действия, работает с файлами или другими артефактами, проверяет промежуточные результаты и возвращает итог.

Для новичков достаточно такого объяснения:

```text
Чат в основном отвечает на вопрос.
Агент помогает выполнить задачу как процесс.
```

## Чат и агент: ключевые отличия

| Критерий | Обычный ИИ-чат | ИИ-агент |
|---|---|---|
| Цель | Ответить на запрос пользователя | Достичь заданной цели через несколько шагов |
| Входные данные | Обычно текст в сообщении | Текст, файлы, задачи, репозиторий, документы, настройки |
| Инструменты | Чаще ограничен диалогом | Может использовать файлы, браузер, терминал, репозиторий, внешние системы, если доступ разрешен |
| Шаги | Пользователь сам ведет диалог | Агент может планировать последовательность действий |
| Работа с файлами | Часто требует ручной загрузки и копирования | Может читать, создавать и редактировать файлы в рабочей среде |
| Самопроверка | Нужно явно попросить | Может быть частью workflow, но результат все равно проверяет человек |
| Контроль человека | Пользователь проверяет ответ | Человек задает границы, подтверждает важные действия и принимает итог |
| Риски | Уверенные ошибки, выдуманные факты | Те же ошибки плюс риск нежелательных действий с файлами, данными или настройками |

## Схема работы агента

Базовая схема:

```text
Цель
  -> входные данные
  -> план
  -> инструменты
  -> действия
  -> самопроверка
  -> ревью человеком
  -> итоговый артефакт
```

### 1. Цель

Цель должна быть конкретной.

Плохо:

```text
Помоги с тестированием.
```

Лучше:

```text
Проанализируй новое требование к форме регистрации, составь вопросы к аналитику и черновик чек-листа.
```

### 2. Входные данные

Агенту нужно дать материал, с которым он работает:

- текст требования;
- ссылку или описание задачи;
- шаблон тест-кейсов;
- предыдущий чек-лист;
- правила команды;
- ограничения по безопасности.

Если входных данных нет, агент будет делать предположения. В учебных материалах такие места нужно явно помечать как «Недостаточно данных» или «Допущение».

### 3. План

План показывает, что агент не просто отвечает одним текстом, а выполняет процесс.

Пример:

1. Прочитать требование.
2. Выделить бизнес-правила.
3. Найти неясности.
4. Сформировать вопросы к аналитику.
5. Предложить чек-лист.
6. Отметить риски и допущения.
7. Передать результат QA на ревью.

### 4. Инструменты

Инструменты должны соответствовать задаче. Добавлять их «для красоты» не нужно.

Для QA-задач возможны:

- документы с требованиями;
- баг-трекер;
- таблицы;
- тест-менеджмент система;
- репозиторий;
- терминал для запуска проверок;
- браузер для просмотра тестового стенда;
- локальные файлы проекта.

### 5. Действия

Действия должны быть безопасными и понятными.

Нормальные действия:

- прочитать файл;
- составить список вопросов;
- создать черновик чек-листа;
- предложить тестовые идеи;
- запустить безопасную проверку;
- суммировать результат.

Опасные действия без подтверждения:

- менять данные в рабочей системе;
- публиковать результат от имени человека;
- удалять файлы;
- отправлять приватные данные во внешний сервис;
- менять статусы задач;
- назначать ответственных.

### 6. Самопроверка

Самопроверка полезна, но не заменяет ревью QA.

Агент может проверить:

- есть ли неподтвержденные допущения;
- есть ли пропущенные негативные сценарии;
- все ли вопросы к требованиям вынесены отдельно;
- нет ли приватных данных;
- можно ли использовать результат как черновик.

### 7. Ревью человеком

Человек должен проверить:

- корректность понимания требований;
- полноту проверок;
- реалистичность рисков;
- безопасность данных;
- применимость результата в проекте.

## Примеры ИИ-агентов для QA

### Анализ требований

**Цель:** каждую неделю разбирать новые требования и готовить вопросы к аналитику.

**Результат:** список бизнес-правил, неясностей, рисков и вопросов.

**Контроль QA:** проверить, что агент не придумал поведение системы.

### Чек-лист по новой функции

**Цель:** подготовить черновик чек-листа по новой фиче.

**Результат:** группы проверок: позитивные, негативные, граничные, интеграционные, регрессионные.

**Контроль QA:** убрать лишние проверки и добавить проектный контекст.

### Черновик тест-кейсов

**Цель:** создать первичный набор тест-кейсов по требованию.

**Результат:** таблица тест-кейсов с шагами и ожидаемыми результатами.

**Контроль QA:** проверить покрытие, корректность expected result и отсутствие выдуманных требований.

### Анализ баг-репортов

**Цель:** сгруппировать баги по компонентам и найти повторяющиеся проблемы.

**Результат:** сводка по компонентам, частым симптомам и вопросам к команде.

**Контроль QA:** проверить, что группировка не скрыла важные отличия между багами.

### PR или изменение в коде

**Цель:** предложить, какие проверки нужны для измененной логики.

**Результат:** список рисков, тестовых идей и областей регрессии.

**Контроль QA:** проверить связь тестовых идей с реальным изменением, а не с общими догадками.

### Еженедельный отчет о качестве

**Цель:** собрать черновик отчета по результатам тестирования.

**Результат:** что проверено, что найдено, какие риски остались, что нужно решить.

**Контроль QA:** проверить факты и не дать агенту сделать выводы без данных.

## Codex как пример агента

Codex удобно показывать как пример агента, который работает не только с сообщением в чате, но и с артефактами проекта.

В рамках такого workflow Codex может:

- читать файлы проекта;
- учитывать инструкции из `AGENTS.md`;
- создавать и редактировать материалы;
- запускать команды проверки;
- смотреть состояние git;
- готовить изменения к коммиту и ревью.

Пример из текущего курса:

```text
Цель:
Подготовить комплект материалов для занятия.

Входные данные:
- ТЗ в Markdown;
- AGENTS.md с правилами;
- уже созданные материалы;
- чек-лист качества.

Действия:
1. Прочитать ТЗ и текущие файлы.
2. Определить недостающие артефакты.
3. Создать следующий файл.
4. Проверить структуру и критерии.
5. Обновить чек-лист.
6. Сделать коммит.

Контроль человека:
Проверить содержание, методическую точность и соответствие задаче.
```

Важно: даже если агент умеет редактировать файлы и запускать команды, человек все равно отвечает за принятие результата.

## Риски агентного подхода

### Риск 1. Агент делает больше, чем нужно

Если цель размыта, агент может создать лишние файлы, изменить не те артефакты или уйти в ненужную детализацию.

Что делать: задавать границы задачи и критерии готовности.

### Риск 2. Агент уверенно развивает неверное допущение

Если входных данных мало, агент может построить длинный workflow на неверной основе.

Что делать: требовать пометки «Недостаточно данных» и «Допущение».

### Риск 3. Агент работает с чувствительными данными

Если дать агенту реальные логи, приватные требования или секреты, появляется риск утечки.

Что делать: обезличивать данные и ограничивать доступ к чувствительным материалам.

### Риск 4. Человек перестает проверять

Агентный результат может выглядеть очень убедительно. Это повышает риск слепого доверия.

Что делать: встраивать ревью человеком как обязательный шаг.

### Риск 5. Агент меняет состояние системы

Если агенту доступны инструменты, он может не только писать текст, но и выполнять действия.

Что делать: опасные действия выполнять только после явного подтверждения человека.

## Практическое задание

Это задание согласовано с практикой 3 из `05_practice_tasks.md`.

### Ситуация

Команда каждую неделю получает новые требования к форме регистрации. QA нужно быстро понять изменения, найти вопросы, подготовить чек-лист и черновик тест-кейсов.

### Задача студента

Заполнить шаблон:

```text
Название агента:

Цель:

Входные данные:

Инструменты:

Шаги работы:
1.
2.
3.

Где нужен контроль человека:

Что агент не имеет права делать:

Критерии готовности результата:
```

### Критерии хорошего ответа

- Цель конкретная и связана с QA-задачей.
- Входные данные перечислены явно.
- Инструменты нужны для задачи, а не добавлены случайно.
- Шаги образуют повторяемый workflow.
- Есть контроль человека.
- Есть запреты по безопасности.
- Результат можно использовать как черновик, а не как финальную истину.

### Типичные ошибки студентов

- Написать один промпт вместо workflow.
- Дать агенту право финально утверждать тест-кейсы.
- Не указать источники требований.
- Не предусмотреть вопросы к аналитику.
- Не указать запрет на приватные данные.
- Считать, что самопроверка агента заменяет ревью QA.

## Короткий вывод для студентов

ИИ-агент полезен, когда задача повторяемая, состоит из нескольких шагов и требует работы с артефактами. Но агент не отменяет ответственность QA. Чем больше действий может выполнить агент, тем важнее границы, проверка и контроль человека.
