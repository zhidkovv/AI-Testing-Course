# Памятка студента

## Главное правило

ИИ делает черновик, QA проверяет.

Ответ нейросети нельзя считать правильным только потому, что он написан уверенно и красиво. Тестировщик проверяет ответ по требованиям, фактам, негативным сценариям и рискам.

Рабочая схема:

```text
Дать контекст -> получить черновик -> проверить -> исправить -> отметить допущения
```

## Где ИИ помогает QA

ИИ может быть полезен, когда нужно:

- объяснить требование простыми словами;
- предложить список проверок;
- составить черновик тест-кейсов;
- сделать чек-лист;
- улучшить баг-репорт;
- предложить тестовые данные;
- найти вопросы к аналитику;
- подсказать идеи для API-проверок;
- объяснить простой SQL-запрос;
- структурировать заметки после тестирования.

ИИ не должен:

- принимать финальное решение вместо QA;
- выдумывать требования;
- подтверждать баг без воспроизведения;
- назначать severity и priority без данных;
- заменять проверку результата человеком.

## Шаблон QA-промпта

```text
Ты помогаешь начинающему QA.

Контекст:
[что тестируем и зачем]

Входные данные:
[требование, баг-репорт, описание API или другой материал]

Задача:
[что нужно получить]

Ограничения:
- не выдумывай требований;
- если данных не хватает, напиши «Недостаточно данных»;
- помечай допущения;
- отделяй факты от предположений;
- добавь чек-лист проверки результата.

Формат ответа:
[таблица, список, тест-кейсы, баг-репорт, вопросы]
```

Пример короткой задачи:

```text
Составь тест-кейсы по требованию. Включи позитивные, негативные и граничные проверки. Не добавляй функции, которых нет в требовании.
```

## Как проверять ответ ИИ

Перед использованием ответа проверь:

- Ответ опирается на исходное требование?
- ИИ не добавил функции, которых нет во входных данных?
- Есть позитивные и негативные сценарии?
- Есть граничные значения, если они важны?
- Ожидаемые результаты можно проверить?
- Есть пометки «Недостаточно данных» там, где информации не хватает?
- Факты отделены от предположений?
- Нет персональных данных, паролей, токенов или приватной информации?
- Результат можно использовать без дополнительной догадки?

Если хотя бы один пункт вызывает сомнение, результат нужно доработать.

## Что нельзя отправлять в ИИ

Не отправляй:

- ФИО, телефоны, email реальных пользователей;
- паспортные данные, адреса, платежные данные;
- пароли, токены, ключи API, cookie, session ID;
- приватные требования компании;
- закрытый исходный код;
- реальные логи с чувствительной информацией;
- скриншоты с персональными или коммерческими данными;
- данные из баг-трекера, если они не обезличены.

Безопасный вариант: заменить реальные данные учебными примерами.

Пример:

```text
Плохо:
Пользователь Ivan Petrov, ivan.petrov@example-company.ru, не может войти.

Лучше:
Тестовый пользователь с валидным email не может войти. Реальный email заменен учебным примером.
```

## Когда данных недостаточно

Если данных не хватает, не нужно придумывать.

Используй такие формулировки:

```text
Недостаточно данных: неизвестен браузер, ОС и версия приложения.
```

```text
Недостаточно данных: точный текст ошибки не указан.
```

```text
Допущение: считаем, что email уже зарегистрирован, потому что это нужно проверить отдельно.
```

```text
Нужны уточнения: повторяется ли ошибка, какой пароль был введен, был ли email новым или уже существующим.
```

Хороший QA не маскирует неизвестные данные красивым текстом.

## Мини-шпаргалка по практикам

### Тест-кейсы

Сначала выпиши свои проверки, потом сравни с ответом ИИ.

Проверь:

- позитивный сценарий;
- негативные сценарии;
- граничные значения;
- точные ожидаемые результаты;
- отсутствие лишних требований.

### Баг-репорт

ИИ может помочь оформить структуру:

- Summary;
- Environment;
- Preconditions;
- Steps to reproduce;
- Actual result;
- Expected result;
- Severity;
- Priority;
- Attachments/logs;
- Missing information;
- Questions to reporter.

Но ИИ не должен выдумывать окружение, точный текст ошибки, severity или priority.

### ИИ-агент

Обычный чат отвечает на запрос.

Агентный workflow включает:

- цель;
- входные данные;
- инструменты;
- шаги;
- ограничения;
- самопроверку;
- контроль человека.

Если в описании есть только один промпт, это еще не агентный workflow.
