# Модуль «Использование ИИ в тестировании»

## Назначение

Этот комплект материалов предназначен для занятия на 90 минут для начинающих тестировщиков ПО. Модуль показывает, как использовать ИИ в ручном тестировании без иллюзии, что нейросеть заменяет мышление QA.

Главная идея занятия: ИИ ускоряет подготовку черновиков, но тестировщик отвечает за постановку задачи, проверку результата, поиск пропусков и безопасность данных.

## Для кого

Аудитория: студенты курса тестировщиков ПО без опыта или с минимальным опытом в IT.

Предполагается, что студенты уже знакомятся или скоро познакомятся с темами:

- роли и процессы разработки;
- Jira или другой баг-трекинг;
- клиент-серверная архитектура, API, HTTP, JSON;
- тест-кейсы и чек-листы;
- баг-репорты;
- тест-дизайн;
- тестирование по спецификации;
- SQL на базовом уровне.

## Что должно получиться у студента

После занятия студент должен уметь:

1. Назвать типовые QA-задачи, где ИИ может помочь.
2. Составить простой QA-промпт с контекстом, входными данными, форматом ответа и ограничениями.
3. Проверить ответ ИИ по чек-листу: полнота, соответствие требованиям, отсутствие выдуманных фактов, покрытие негативных сценариев.
4. Улучшить черновик тест-кейсов или баг-репорта с помощью ИИ.
5. Объяснить, чем ИИ-агент отличается от обычного чата.
6. Назвать базовые риски: персональные данные, коммерческая тайна, приватные требования, исходный код, токены и пароли.

## Методический принцип

Не показывать ИИ как «волшебную кнопку». Каждый пример должен включать не только генерацию ответа, но и его проверку.

Рабочая формула занятия:

```text
Контекст + входные данные + ограничение + формат ответа + проверка QA = полезный результат
```

Если в ответе ИИ есть уверенная формулировка без источника или без опоры на требования, студент должен считать это гипотезой, а не фактом.

## Ограничения

- Нельзя требовать от студентов покупки платной подписки.
- Нельзя загружать в ИИ реальные персональные данные, пароли, токены, приватные логи, коммерческие требования или закрытый исходный код.
- Нельзя заполнять недостающие факты в баг-репорте или требованиях без явной пометки «допущение».
- Нельзя сравнивать ИИ-сервисы по ценам и лимитам без свежей проверки официальных источников.

## Что проверить перед занятием

1. Есть ли у студентов доступ к интернету.
2. Можно ли использовать бесплатный ИИ-чат в аудитории.
3. Есть ли у преподавателя доступ к продвинутому ИИ-инструменту для демонстрации.
4. Готовы ли офлайн-заготовки ответов ИИ на случай, если сервисы недоступны.
5. Понятны ли студентам базовые термины: тест-кейс, баг-репорт, expected result, actual result, severity, priority.

## Рекомендуемый способ проведения

Преподаватель не должен читать длинную лекцию. Лучше идти короткими циклами:

1. Объяснить задачу.
2. Показать пример промпта.
3. Получить или показать ответ ИИ.
4. Вместе найти ошибки и пропуски.
5. Зафиксировать правило проверки.

Такой формат лучше соответствует начинающей аудитории и снижает риск, что занятие превратится в обзор популярных сервисов вместо тренировки QA-навыка.
