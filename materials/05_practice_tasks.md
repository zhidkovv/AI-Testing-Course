# Практические задания

## Общий принцип

Каждая практика построена по одной схеме:

1. Студент сначала думает сам.
2. Затем использует ИИ как помощника.
3. Потом проверяет ответ ИИ.
4. Итоговый результат принимает не ИИ, а студент.

Практики можно проводить с любым доступным ИИ-чатом. Если у студентов нет доступа к сервисам, преподаватель использует заранее подготовленные ответы ИИ и дает студентам проверить их как артефакты.

---

## Упражнение 1. ИИ генерирует тест-кейсы, QA проверяет

### Цель

Показать, что ИИ быстро предлагает тесты, но может пропустить важные проверки, добавить лишние предположения или плохо покрыть негативные сценарии.

### Длительность

18 минут.

### Входные данные

Мини-спецификация формы регистрации:

```text
Пользователь может зарегистрироваться по email и паролю.

Email обязателен и должен быть валидным.

Пароль обязателен, длина от 8 до 32 символов, должен содержать хотя бы одну букву и одну цифру.

После успешной регистрации пользователь видит сообщение «Регистрация завершена».

Если email уже используется, показывается ошибка «Email уже зарегистрирован».
```

### Инструкция для студентов

1. За 3 минуты самостоятельно выпишите 5-7 проверок.
2. Составьте промпт для ИИ по шаблону ниже.
3. Получите или прочитайте ответ ИИ.
4. Сравните ответ с собственным списком.
5. Отметьте:
   - пропущенные проверки;
   - дубли;
   - проверки, которых нет в требованиях;
   - неясные формулировки;
   - отсутствие негативных сценариев.
6. Составьте итоговый набор из 8-12 проверок.

### Промпт-шаблон

```text
Ты помогаешь начинающему QA составить тест-кейсы.

Контекст:
Нужно протестировать форму регистрации.

Требование:
[вставь мини-спецификацию]

Задача:
Составь набор тест-кейсов. Включи позитивные и негативные проверки.

Ограничения:
- не выдумывай требований, которых нет в спецификации;
- если данных не хватает, укажи это отдельно;
- добавь проверки граничных значений для пароля;
- не добавляй проверки интерфейса, если они не следуют из требования.

Формат:
Таблица с колонками: ID, Название, Предусловия, Шаги, Ожидаемый результат, Тип проверки.

В конце добавь чек-лист, по которому QA должен проверить твой ответ.
```

### Ожидаемый результат

У студентов должен получиться набор проверок, который покрывает минимум:

- успешную регистрацию с валидным email и паролем;
- пустой email;
- невалидный email;
- пустой пароль;
- пароль короче 8 символов;
- пароль ровно 8 символов;
- пароль ровно 32 символа;
- пароль длиннее 32 символов;
- пароль без буквы;
- пароль без цифры;
- email, который уже используется;
- текст сообщения при успешной регистрации;
- текст ошибки для занятого email.

### Пример итогового набора проверок

| ID | Проверка | Данные | Ожидаемый результат |
|---|---|---|---|
| TC-01 | Успешная регистрация | `new_user@example.com`, `pass1234` | Показано сообщение «Регистрация завершена» |
| TC-02 | Email пустой | пустой email, `pass1234` | Регистрация не выполнена, показана ошибка обязательности email |
| TC-03 | Email невалидный | `userexample.com`, `pass1234` | Регистрация не выполнена, показана ошибка формата email |
| TC-04 | Пароль пустой | `user@example.com`, пустой пароль | Регистрация не выполнена, показана ошибка обязательности пароля |
| TC-05 | Пароль 7 символов | `user@example.com`, `pass123` | Регистрация не выполнена |
| TC-06 | Пароль 8 символов | `user@example.com`, `pass1234` | Регистрация выполнена |
| TC-07 | Пароль 32 символа | валидный email, пароль 32 символа с буквой и цифрой | Регистрация выполнена |
| TC-08 | Пароль 33 символа | валидный email, пароль 33 символа | Регистрация не выполнена |
| TC-09 | Пароль без цифры | `user@example.com`, `password` | Регистрация не выполнена |
| TC-10 | Пароль без буквы | `user@example.com`, `12345678` | Регистрация не выполнена |
| TC-11 | Email уже используется | существующий email, валидный пароль | Показана ошибка «Email уже зарегистрирован» |

### Критерии проверки

- Есть позитивные и негативные проверки.
- Есть граничные значения пароля: 7, 8, 32, 33 символа.
- Проверки не требуют функций, которых нет в спецификации.
- Ожидаемые результаты сформулированы проверяемо.
- Отдельно отмечено, каких данных не хватает: например, точные тексты ошибок для пустого email и невалидного пароля.
- Итоговый набор не является слепой копией ответа ИИ.

### Типичные ошибки

- Проверять имя пользователя, телефон или капчу, хотя их нет в спецификации.
- Не проверить границы 8 и 32 символа.
- Проверить только позитивный сценарий.
- Написать «должна появиться ошибка» без конкретики, где конкретика есть в требовании.
- Принять ответ ИИ без сравнения с требованиями.

### Вариант усложнения

Добавить требование:

```text
После трех неуспешных попыток регистрации с одного IP форма блокируется на 10 минут.
```

Студенты должны добавить проверки счетчика попыток, блокировки и восстановления доступа.

### Вариант без платного ИИ-сервиса

Преподаватель заранее готовит один неполный ответ ИИ. Студенты не генерируют ответ сами, а проводят ревью готового результата и исправляют его.

## Упражнение 2. Плохой баг-репорт -> хороший баг-репорт

### Цель

Показать, как ИИ помогает структурировать баг-репорт, но не должен придумывать факты, которых нет во входных данных.

### Длительность

12 минут.

### Входные данные

Плохой баг-репорт:

```text
Не работает регистрация. Я ввела почту и пароль, нажала кнопку, что-то пошло не так. Исправьте.
```

### Инструкция для студентов

1. Сначала выпишите, каких данных не хватает.
2. Составьте промпт для улучшения баг-репорта.
3. Получите или прочитайте ответ ИИ.
4. Найдите места, где ИИ выдумал факты.
5. Перепишите результат так, чтобы неизвестные поля были явно помечены.
6. Составьте вопросы к автору бага.

### Промпт-шаблон

```text
Ты помогаешь начинающему QA улучшить баг-репорт.

Исходное описание:
[вставь плохой баг-репорт]

Задача:
Преобразуй описание в структурированный баг-репорт.

Ограничения:
- не выдумывай факты;
- если данных нет, напиши «Недостаточно данных»;
- отдели факты от предположений;
- составь список уточняющих вопросов.

Формат:
- Summary
- Environment
- Preconditions
- Steps to reproduce
- Actual result
- Expected result
- Severity
- Priority
- Attachments/logs
- Missing information
- Questions to reporter
```

### Ожидаемый результат

Структурированный баг-репорт, где неизвестные данные не подменены догадками.

Пример:

```text
Summary:
Не удалось завершить регистрацию после ввода email и пароля.

Environment:
Недостаточно данных: неизвестны устройство, ОС, браузер, версия приложения, стенд.

Preconditions:
Недостаточно данных: неизвестно, был ли email новым или уже зарегистрированным.

Steps to reproduce:
1. Открыть форму регистрации.
2. Ввести email.
3. Ввести пароль.
4. Нажать кнопку регистрации.

Actual result:
Со слов автора: «что-то пошло не так». Точный текст ошибки или поведение неизвестны.

Expected result:
Если email и пароль валидны, пользователь должен увидеть сообщение «Регистрация завершена».

Severity:
Недостаточно данных. Возможная severity зависит от воспроизводимости и охвата проблемы.

Priority:
Недостаточно данных. Priority должен определить владелец продукта или команда.

Attachments/logs:
Недостаточно данных. Нужны скриншот, видео, консольные ошибки или логи, если доступны.

Missing information:
- точный email вводить не нужно, но нужен признак: новый или уже зарегистрированный;
- формат пароля: длина, есть ли буква и цифра;
- точный текст ошибки;
- браузер/ОС/устройство;
- повторяется ли ошибка;
- дата и время попытки;
- стенд или окружение.

Questions to reporter:
1. Какой точный текст ошибки появился?
2. Email был новым или уже зарегистрированным?
3. Пароль соответствовал требованиям?
4. В каком браузере и на каком устройстве воспроизводится проблема?
5. Проблема повторяется при новой попытке?
```

### Критерии проверки

- Баг-репорт стал структурированным.
- Недостающие данные не выдуманы.
- Есть шаги воспроизведения, даже если они неполные.
- Actual result отделен от expected result.
- Severity и priority не назначены уверенно без данных.
- Есть список уточняющих вопросов.
- Нет персональных данных в примерах.

### Типичные ошибки

- Написать конкретный браузер или устройство, которых нет во входных данных.
- Указать severity `Critical` без оценки влияния.
- Придумать точный текст ошибки.
- Использовать реальный email пользователя.
- Не задать уточняющие вопросы.

### Вариант усложнения

Добавить фрагмент лога:

```text
POST /api/register returned 500
```

Студенты должны дополнить баг-репорт технической информацией, но не делать вывод о root cause без доказательств.

### Вариант без платного ИИ-сервиса

Преподаватель дает два варианта ответа ИИ:

- первый слишком уверенно выдумывает окружение и severity;
- второй аккуратно помечает неизвестные данные.

Студенты сравнивают варианты и объясняют, какой лучше для реального баг-трекинга.

## Упражнение 3. Спроектируй ИИ-агента для QA-задачи

### Цель

Закрепить отличие ИИ-агента от обычного чата и показать, где в агентном workflow нужен контроль человека.

### Длительность

10 минут.

### Входные данные

Ситуация:

```text
Команда каждую неделю получает новые требования к форме регистрации.

Нужно анализировать требования, предлагать чек-лист проверок, находить риски, формировать черновики тест-кейсов и список вопросов к аналитику.
```

### Инструкция для студентов

1. Определите цель агента.
2. Определите входные данные.
3. Определите инструменты, которые агенту могут понадобиться.
4. Опишите шаги работы агента.
5. Отметьте, где человек должен проверить результат.
6. Сформулируйте критерии готовности результата.

### Шаблон workflow

```text
Название агента:

Цель:

Входные данные:

Инструменты:

Шаги работы:
1.
2.
3.

Где нужен контроль человека:

Что агент не имеет права делать:

Критерии готовности результата:
```

### Промпт-шаблон

```text
Ты помогаешь спроектировать ИИ-агента для QA-задачи.

Ситуация:
[вставь описание задачи]

Задача:
Опиши агентный workflow, который поможет QA каждую неделю анализировать новые требования.

Ограничения:
- агент не принимает финальные решения вместо QA;
- агент не отправляет приватные данные во внешние сервисы;
- агент помечает допущения;
- агент формирует вопросы, если требований недостаточно.

Формат:
Используй шаблон: цель, входные данные, инструменты, шаги, контроль человека, запреты, критерии готовности.
```

### Ожидаемый результат

Пример заполненного workflow:

```text
Название агента:
QA Requirements Assistant

Цель:
Помочь QA быстро разобрать новые требования к форме регистрации и подготовить черновик тестовых артефактов.

Входные данные:
- текст новых требований;
- ссылка на задачу в баг-трекере;
- предыдущий чек-лист по форме регистрации;
- шаблон тест-кейсов команды.

Инструменты:
- документы с требованиями;
- баг-трекер;
- таблица или тест-менеджмент система;
- репозиторий или документация API, если доступ разрешен;
- локальные файлы проекта, если это безопасно.

Шаги работы:
1. Прочитать новое требование.
2. Выделить бизнес-правила и ограничения.
3. Найти неясности и противоречия.
4. Составить вопросы к аналитику.
5. Предложить чек-лист проверок.
6. Сформировать черновики тест-кейсов.
7. Отметить допущения и риски.
8. Передать результат QA на проверку.

Где нужен контроль человека:
- подтверждение понимания требований;
- выбор финальных тестов;
- приоритизация рисков;
- отправка вопросов аналитику;
- внесение тест-кейсов в рабочую систему.

Что агент не имеет права делать:
- публиковать результат без проверки QA;
- выдумывать требования;
- загружать приватные данные во внешние сервисы;
- менять задачи в баг-трекере без подтверждения;
- назначать ответственных без согласования.

Критерии готовности результата:
- список вопросов к требованиям;
- чек-лист покрывает позитивные, негативные и граничные сценарии;
- тест-кейсы помечены как черновик;
- все допущения явно указаны;
- QA может проверить результат за разумное время.
```

### Чем это отличается от обычного чата

Обычный чат чаще отвечает на один запрос. Агентный workflow описывает повторяемый процесс:

- есть цель;
- есть входные данные;
- есть инструменты;
- есть последовательность шагов;
- есть ограничения;
- есть самопроверка;
- есть точки контроля человека.

### Критерии проверки

- Цель агента сформулирована конкретно.
- Входные данные перечислены явно.
- Инструменты связаны с задачей, а не добавлены «для красоты».
- Есть пошаговый процесс.
- Есть точки контроля человека.
- Есть запреты по безопасности.
- Результат можно использовать повторно каждую неделю.

### Типичные ошибки

- Описать один промпт вместо workflow.
- Дать агенту право принимать финальное решение вместо QA.
- Не указать, откуда агент получает требования.
- Не предусмотреть проверку человеком.
- Не обозначить запрет на приватные данные.

### Вариант усложнения

Добавить условие:

```text
Требования приходят из двух источников: задача в Jira и комментарии аналитика в документе. Иногда они противоречат друг другу.
```

Студенты должны добавить шаг поиска противоречий и правило: агент не выбирает верную версию сам, а формирует вопрос ответственному человеку.

### Вариант без платного ИИ-сервиса

Студенты проектируют workflow на бумаге или в Markdown без генерации ИИ. После этого преподаватель показывает пример агентного workflow и группа сравнивает решения.
